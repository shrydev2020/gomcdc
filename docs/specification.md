# gomcdc Normative Specification

This document is the single normative specification for `gomcdc`.

The earlier requirements, amendments, and emergency corrections cease to be normative once incorporated into this document.

The Japanese document is normative, and this English document is an equivalent translation.

The specification version is `1.0-draft`.

The JSON schema and CLI have no compatibility guarantee before the `1.0` release.

## 1. Purpose

`gomcdc` measures the following metrics from tests of a Go module and integrates them into one report.

- Statement Coverage
- Function Coverage
- Decision Coverage
- Switch Clause Body Coverage
- Type Switch Clause Body Coverage
- Select Clause Body Coverage
- Switch Clause Selection Coverage
- Type Switch Clause Selection Coverage
- Condition Coverage
- Unique-Cause MC/DC
- Masking MC/DC
- Instrumentation Coverage for each metric

The tool must not guess unavailable facts and report them as covered or not-covered.

## 2. Supported environment

The module language version is Go 1.26.

Development and CI use the latest security-patched release in the Go 1.26 line.

Go Modules, Linux, macOS, and GitHub Actions are supported.

The compiler-aware backend declares each supported Go compiler version and reports unsupported-by-backend instead of guessing on another version.

Windows, assembly, cgo internals, compiler IR, path coverage, and distributed test execution are outside version 1.

## 3. Terms

A **Condition** is an atomic source expression of type bool that contains none of `&&`, `||`, or `!`.

Boolean results of comparisons, function calls, field selections, and map lookups are not decomposed further.

A **Condition occurrence** is one source occurrence of a condition.

Repeated source text receives a separate index for each occurrence.

A **Decision** is a complete Boolean expression that selects control flow.

A **Clause body** is the statement list belonging to a case or default clause of a switch, type switch, or select.

An **Evaluation trace** is the condition-state vector and decision result observed during one decision evaluation.

A condition state is `true`, `false`, or `not-evaluated`.

## 4. Source scope

Only packages in the main module returned by `go list` for the supplied Go package patterns are measured.

The standard library, external modules, vendor, and sources generated by this tool are excluded.

User sources carrying the standard Go generated-code comment are excluded by default.

`_test.go` files enter AST metric denominators only when `--include-tests` is set.

Because standard Go coverage excludes test sources from its statement denominator, `--include-tests` does not affect Statement or Function Coverage.

Analysis and `go test` must use the same GOOS, GOARCH, build tags, CGO settings, and module settings.

## 5. Statement Coverage

Statement Coverage comes from a standard Go cover profile run against the original source.

The measurement unit is the statement count attached to a Go cover-profile block.

When a block counter is greater than zero, all statement units in that block are covered.

The numerator is the number of covered statement units.

The denominator is the sum of statement counts in target blocks.

Detailed reports present a statement region with its original source range, statement count, and counter instead of inventing individual AST-statement evidence.

## 6. Function Coverage

Functions, methods, and function literals are Function Coverage entities.

A function is covered when at least one statement unit belonging to it is covered.

Functions without statement units are excluded from the denominator.

Multiple `init` functions and function literals are distinguished by original source location.

## 7. Decision Coverage

Version 1 recognizes only these decisions.

- The condition of an `if` statement
- The condition of a conditional `for` statement
- Each case expression of a conditionless switch

Boolean expressions used only as assignments, return values, or call arguments are not decisions.

When one conditionless-switch case clause contains multiple expressions, each expression is an independent decision.

Case expressions after an earlier true expression are `not-evaluated`, not false.

Decision Coverage must not be called CFG Edge Coverage.

Edges for `goto`, labeled break, labeled continue, return, panic, recover, and fallthrough are excluded from Decision Coverage.

Each decision has two coverage obligations, the true outcome and the false outcome.

The numerator is the number of observed outcomes.

The denominator is two.

Only completed evaluations provide evidence.

## 8. Condition expression tree

A decision is normalized into this expression tree.

```go
type DecisionExpr interface { isDecisionExpr() }
type AtomExpr struct { ConditionIndex uint16 }
type NotExpr struct { Operand DecisionExpr }
type AndExpr struct { Left, Right DecisionExpr }
type OrExpr struct { Left, Right DecisionExpr }
```

Parentheses affect tree structure but do not introduce conditions.

For `!a`, the condition is `a`, and the tree is `NotExpr(AtomExpr(a))`.

For `a && (b || c)`, source occurrence order deterministically assigns `a=0`, `b=1`, and `c=2`.

Condition indexes must not depend on file loading, map iteration, goroutine scheduling, or parallel analysis order.

## 9. Condition Coverage

Each condition occurrence has two coverage obligations, true and false.

A condition skipped by short-circuit evaluation is recorded as `not-evaluated`.

`not-evaluated` must not be treated as false.

`not-evaluated` does not add an obligation to the denominator.

Only true and false values actually observed in completed evaluations provide evidence.

## 10. Evaluation lifecycle

Every dynamic decision evaluation receives an EvaluationID.

EvaluationID must be unique within one test process.

The module-wide identity is:

```go
type EvaluationIdentity struct {
    RunID        string
    PackagePath  string
    ProcessID    int
    EvaluationID uint64
}
```

The runtime API explicitly represents Begin, condition recording, End, and Abort.

Only an evaluation that reaches End is completed.

An evaluation that does not reach End because of panic, `runtime.Goexit`, process interruption, or another termination is aborted.

Aborted evaluations cannot prove Decision, Condition, or MC/DC coverage.

Recursion, loop iterations, nested decisions, goroutines, and test processes must not mix evaluations.

## 11. Common MC/DC rules

An MC/DC percentage in this tool is the coverage of independent-effect obligations for condition occurrences.

A complete MC/DC claim also requires the corresponding Decision and Condition Coverage obligations to be satisfied.

The percentage does not include complete entry and exit invocation, and the tool does not claim DO-178C compliance, tool qualification, or safety certification.

### 11.1 Unique-Cause MC/DC

A target condition `i` is covered when two completed evaluations `p` and `q` satisfy all of these rules.

1. `p[i]` and `q[i]` were both evaluated.
2. `p[i] != q[i]`.
3. `p.result != q.result`.
4. For every `j != i`, `p[j] == q[j]`.
5. `not-evaluated` is unequal to true and false.

Rule 4 compares condition states, not guessed input values.

When short-circuit semantics make such a pair impossible and the expression tree proves that no Unique-Cause witness pair exists, the status is `infeasible`.

`infeasible` does not claim general program-path unreachability.

When impossibility cannot be proved, the status is `not-covered`.

For `a && b` with only these traces, Unique-Cause results are `a=infeasible` and `b=covered`.

```text
[false, not-evaluated] -> false
[true,  true]          -> true
[true,  false]         -> false
```

## 12. Masking MC/DC

Masking MC/DC is analyzed for each condition occurrence.

Analysis must be a pure function of DecisionExpr and completed evaluations.

A complete assignment assigns true or false to every condition.

A completion of observed trace `p` agrees with all evaluated states, produces `p.result` under Go left-to-right short-circuit evaluation, and leaves the same conditions not-evaluated.

Condition `j` is masked under complete assignment `x` when flipping only `j` preserves the decision result.

```text
eval(tree, x) == eval(tree, flip(x, j))
```

A target condition `i` is covered when completed evaluations `p` and `q` and completions `x` and `y` satisfy all of these rules.

1. `p[i]` and `q[i]` were both evaluated.
2. `x[i] != y[i]`.
3. `eval(tree, x) != eval(tree, y)`.
4. For every `j != i`, either `x[j] == y[j]`, or `j` is masked in both `x` and `y`.
5. `x` and `y` are valid completions of `p` and `q` respectively.

A witness stores observed traces, completions, the target condition, and masked conditions.

Repeated source expressions are separate condition occurrences, and the analyzer must not assume their values are coupled.

When a witness would require an unproved semantic coupling between repeated occurrences, the status is `analysis-incomplete`.

When an exact search cannot complete because of a resource limit, the status is also `analysis-incomplete`, and no guessed score is returned.

For the three `a && b` traces above, Masking MC/DC results are `a=covered` and `b=covered`.

## 13. Clause Body Coverage

The AST backend measures body entry, not case selection.

These three metrics are aggregated independently.

- Switch Clause Body Coverage
- Type Switch Clause Body Coverage
- Select Clause Body Coverage

Each case or default body contributes one coverage obligation.

A body is covered when execution enters it.

An explicitly written case or default belongs in the denominator even when its original body is empty.

A body reached through `fallthrough` is covered for body coverage.

Body execution must not be called direct case selection.

A conditionless-switch case expression is a Decision Coverage entity, and its body is also a Switch Clause Body Coverage entity.

### 13.1 Clause Selection Coverage

A compiler-aware backend measures Switch Clause Selection Coverage and Type Switch Clause Selection Coverage.

Each explicit case, each default, and the no-match outcome when no default exists contributes one selection obligation.

Only a directly selected clause is covered.

A clause body reached through fallthrough is not covered as a direct selection.

`case A, B` contributes one clause-selection obligation in version 1.

Compiler events retain enough evidence to distinguish the matched case expression or matched type alternative.

A selection event contains SwitchID and the directly selected ClauseID and contains a matched-alternative index only when a case alternative exists.

A no-match event contains SwitchID and a no-match role and does not invent a nonexistent ClauseID.

A fallthrough event contains source and destination ClauseIDs.

Matched alternatives do not enter version 1 coverage denominators, but reports must not discard collected evidence.

A fallthrough edge is retained as an independent event and is not mixed into selection or body-entry metrics.

Switch Clause Selection Coverage and Type Switch Clause Selection Coverage are version 1 completion requirements.

No generic `Clause Coverage` aggregate is defined.

## 14. Instrumentation Coverage

Coverage scores and instrumentation support are separate.

Every metric reports:

```text
discovered
supported
instrumented
unsupported
unknown
```

`discovered` is the number of entities statically identified in target source.

`supported` is the number for which the selected backend can produce exact evidence.

`instrumented` is the number for which the required probe or compiler event was actually produced.

`unsupported` is the number proven outside backend capability.

`unknown` is the number for which analysis or integrity validation could not complete and capability could not be determined.

The unit is the source entity carrying an obligation for that metric.

Statement uses statement units, Function uses functions, Decision uses decisions, Condition and both MC/DC metrics use condition occurrences, Clause Body metrics use clause bodies, and Clause Selection metrics use selection obligations.

Instrumentation Coverage divides instrumented by discovered.

Support status is one of `supported`, `unsupported-by-backend`, and `unknown`.

Coverage status is one of `covered`, `not-covered`, `infeasible`, and `analysis-incomplete`.

`aborted` is an evaluation status and is not converted into a coverage-entity status.

Unsupported-by-backend and unknown entities are excluded from coverage-score denominators.

No setting mixes unsupported or unknown entities into coverage percentages.

`--strict` fails when a requested metric has any unsupported-by-backend, unknown, analysis-incomplete, or uninstrumented entity.

## 15. Semantic preservation

Instrumentation preserves:

- Go-specified evaluation order
- `&&` and `||` short-circuit behavior
- expression and function-call evaluation counts
- panic conditions
- defer order
- side-effect order
- return values and errors
- state changes observable by user code

Execution time, allocation counts, complete stack traces, inlining, goroutine scheduling, CPU usage, and memory usage are not preserved.

The runtime receives each user expression value once and returns the same value.

A runtime instrumentation failure must not propagate to user code as a panic.

Instrumentation failures are reflected in diagnostics and integrity status.

## 16. Source locations and IDs

The authoritative user-facing location is a physical path relative to the main module root and physical one-based line and column positions.

Logical positions from user `//line` directives may be retained as additional information but do not replace the authoritative location.

Reports must not expose temporary-workspace, generated-bridge, or injected-runtime locations as user-source locations.

DecisionID and ClauseID must be deterministic for one source revision.

IDs may use module path, package import path, relative file path, node kind, physical start offset, physical end offset, and condition index.

IDs must not depend on load order, analysis order, goroutine order, or map iteration.

Stability across source revisions is not guaranteed.

## 17. Evidence authority and measurement modes

Only the standard-cover backend owns execution evidence for Statement and Function Coverage.

Only the AST backend owns execution evidence for Decision, Condition, both MC/DC metrics, and the three Clause Body metrics.

Only the compiler-aware backend owns execution evidence for Switch and Type Switch Clause Selection.

A backend must not guess or derive another backend’s evidence and promote it to authoritative evidence.

### 17.1 Measurement modes

Selecting only Statement or Function metrics uses `standard-cover`.

Selecting only non-standard-cover metrics uses `single-run`.

Selecting both groups uses `dual-run-standard-cover`.

Dual-run mode executes one standard-cover run on original source and one instrumented run on instrumented source.

The instrumented run applies the AST and compiler-aware backends to the same build and test execution.

Selecting Clause Selection metrics must not add a third test run.

Tests must not rerun once per metric.

When a selected metric requires metadata or runtime evidence also used by another metric, the implementation may collect that dependency.

Collecting dependency data must not enable an unselected metric in the report.

The two runs are separate test executions, and their evaluations must not be correlated.

Each run reports its own status, failure kind, and package status.

When one run fails, the tool still attempts to collect the other run and all valid partial evidence from the failed run.

## 18. `go test` arguments

At least one package pattern is required.

Go build and test arguments after `--` apply with the same meaning to analysis and every measurement run.

The tool always sets `-count=1`.

A different `-count` supplied directly or through GOFLAGS is a CLI error.

User-supplied `-cover`, `-covermode`, `-coverpkg`, `-coverprofile`, `-json`, and `-overlay` are CLI errors because the tool owns them.

Non-conflicting arguments such as `-run`, `-skip`, `-tags`, `-race`, `-shuffle`, `-parallel`, and `-timeout` are preserved.

A go.work file with multiple active main modules is unsupported, and the tool must not guess one module.

## 19. Runtime data

Each test process writes to an independent event file.

File identity includes a collision-resistant encoding of package import path, PID, RunID, and a nonce.

A package path must not be concatenated directly as a file-system path component.

After a truncated tail, panic, timeout, or abnormal exit, the collector recovers every fully validated record.

It validates DecisionID, EvaluationIdentity, condition count, condition states, result, and status.

A completed vector inconsistent with expression-tree short-circuit semantics is removed from evidence and reported as an integrity error.

Exact duplicates of decision, vector, result, and status may be aggregated.

Aggregation must retain a representative vector, occurrence count, and all information needed by witnesses.

TestID is best-effort information and is `unknown` when unavailable.

TestID is not an MC/DC criterion.

## 20. CLI

The basic invocation is:

```sh
gocoverage test ./...
```

`--coverage` accepts only these canonical names.

```text
statement
function
decision
switch-clause-body
type-switch-clause-body
select-clause-body
switch-clause-selection
type-switch-clause-selection
condition
mcdc-unique
mcdc-masking
all
```

Aliases such as `c0`, `c1`, `c2`, and `mcdc` are not provided.

Primary options are:

```text
--format text|json
--output <path>
--exclude <module-relative-glob>
--include-tests
--keep-workdir
--workdir <parent>
--timeout <duration>
--strict
```

`--exclude` is repeatable, and an empty or unparseable glob is an error.

Output goes to stdout when `--output` is absent.

The temporary workspace is removed at exit unless `--keep-workdir` is set.

The default is `all`.

`all` selects all eleven canonical metrics other than `all` itself.

Threshold options are:

```text
--fail-under-statement
--fail-under-function
--fail-under-decision
--fail-under-switch-clause-body
--fail-under-type-switch-clause-body
--fail-under-select-clause-body
--fail-under-switch-clause-selection
--fail-under-type-switch-clause-selection
--fail-under-condition
--fail-under-mcdc-unique
--fail-under-mcdc-masking
```

A threshold for a metric absent from `--coverage` is a CLI error.

There is no generic `--fail-under-clause`.

## 21. Coverage ratios and thresholds

A coverage ratio divides covered obligations by the sum of covered and not-covered obligations.

Unsupported-by-backend, unknown, infeasible, and analysis-incomplete items are excluded from coverage-score denominators.

When the denominator is zero, the percentage is mathematically undefined and is rendered as `n/a` in text and `null` in JSON.

A configured threshold for a zero-denominator metric is not satisfied.

Threshold comparison uses the unrounded integer ratio.

Displayed percentages are rounded to two decimal places.

## 22. Exit status

Each measurement run distinguishes passed, test-failed, build-failed, timeout, and command-failed.

The overall result retains all run, integrity, strict, and threshold outcomes.

Process exit codes are:

```text
0 success
1 one or more go test runs failed
2 measurement, instrumentation, integrity, or report failure
3 coverage threshold failure
4 invalid CLI usage
```

When conditions overlap, priority is `4 > 2 > 1 > 3 > 0`.

The tool emits every report it can build after a test or measurement failure.

## 23. JSON report

Reports aggregate by module, package, file, function, decision, condition, clause body, and clause selection.

A parent numerator and denominator are calculated from summed obligations, not an average of child percentages.

The root `version` is `1.0-draft` during the draft period and becomes `1` at the 1.0 release.

The JSON root contains at least:

```text
version
module
run
measurementMode
measurements
instrumentationCoverage
summary
packages
errors
```

Summary keys are lower-camel-case forms of canonical CLI metrics.

```text
statement
function
decision
switchClauseBody
typeSwitchClauseBody
selectClauseBody
switchClauseSelection
typeSwitchClauseSelection
condition
mcdcUnique
mcdcMasking
```

Generic `clause` and `clauseBody` keys are not emitted.

IDs are hexadecimal strings, not JSON numbers.

Enum values are listed by the schema and are never coerced to known values.

Arrays are deterministically ordered by package path, physical source location, and ID.

Decision reports include expression, location, outcomes, conditions, both MC/DC results, evaluation vectors, and witnesses.

Each error contains at least `phase`, `code`, and `message`, and contains `package` and a module-relative `path` only when applicable.

## 24. Text report

Every enabled metric displays numerator, denominator, and percentage or `n/a`.

Instrumentation Coverage and unsupported, unknown, and analysis-incomplete counts appear next to coverage scores.

MC/DC witnesses display observed vectors, decision results, and completions used for Masking MC/DC.

When Unique-Cause and Masking results differ, the report identifies the unsatisfied rule.

## 25. Performance and resource boundaries

Within one measurement run, tests and builds must not repeat in proportion to the number of decisions or metrics.

One instrumented run performs source instrumentation, compiler instrumentation, build, and test once to collect every selected non-standard-cover metric.

Duplicate evaluation evidence is aggregated when doing so preserves witness information.

The event journal is compacted periodically and must not retain every loop record in memory.

Performance goals are at most twice ordinary `go test` time for AST-only measurement and at most five times when both MC/DC metrics are selected, but test-dependent timing is not a conformance guarantee.

## 26. Security and trust boundary

The tool builds and runs the target module with the current user's permissions.

The target module must be trusted.

The temporary workspace is not a security sandbox.

Target tests can access event files and environment variables, so evidence authenticity is not guaranteed against malicious target code.

Temporary directories and event files use permissions writable and readable only by the current user.

Source copying prevents symlinks and hard links from writing outside the workspace.

Reports may contain module-relative paths, source expressions, and test output and are treated as project data.

Normal reports must not expose the user's absolute local paths.

## 27. Acceptance criteria

Integration fixtures cover at least:

- AND, OR, NOT, and nested expressions
- `a && b` with `a=infeasible` and `b=covered` for Unique-Cause, and both covered for Masking
- side effects and evaluation order
- panic, recover, defer, and `runtime.Goexit`
- recursion, loops, and nested decisions
- goroutines and parallel package execution
- multiple conditionless-switch case expressions
- expression switch, type switch, select, empty bodies, direct selection, no-match, and fallthrough
- multiple packages, external test packages, and build tags
- test failure, build failure, timeout, and truncated event files
- source mapping, user `//line`, and generated-code exclusion
- dual-run execution and partial recovery
- all thresholds and exit-code precedence

These commands must pass.

```sh
go test -count=1 ./...
go test -count=1 -race ./...
go vet ./...
```

## 28. Completion

One `gocoverage test ./...` invocation must emit the selected metrics, Instrumentation Coverage, source locations, and partial-run status in one report.

Completion requires both that the AST backend never reports unavailable selection information as covered or not-covered and that the compiler-aware backend reports Clause Selection Coverage exactly.

All acceptance fixtures, race tests, and `go vet` must pass.
